# 侧边菜单页面图标统一设计

日期：2026-05-12

## 背景

当前产品侧边菜单已经有一套基于 `entry.key -> .icon-*` 的纯 CSS 图标机制，但只有部分页面项具备图标语义。像 `账套管理`、`部门管理`、`考勤数据查询` 这样的页面可以显示图标，其他页面项则存在缺失、语义不完整或风格不够统一的问题。

本次目标不是更换图标技术栈，而是在现有菜单系统上补齐所有页面项的图标，并统一为一套稳定的企业后台线性描边风格。

## 目标

- 侧边菜单所有页面项都显示与业务相符的小图标。
- 图标统一为线性描边风格，尺寸、线宽、重心和激活态表现保持一致。
- 不引入第三方图标库，不改动后端导航结构，不调整菜单文案。
- 保持现有模板渲染结构，继续使用 `entry.key` 驱动图标类名。

## 非目标

- 不重做顶部模块导航图标。
- 不修改移动端底部模块导航。
- 不切换到 SVG 图标方案。
- 不顺带调整菜单文案、信息架构或权限逻辑。

## 设计方案

### 1. 保持现有模板结构

继续使用 [templates/partials/app_nav.html](/Users/lewis/Lewis/code/git/MtEmployeeAttendance-System/templates/partials/app_nav.html) 现有结构：

- 菜单项继续输出 `app-side-icon icon-{{ entry.key|replace('_', '-') }}`
- 不新增模板分支，不引入额外映射表
- 图标命名统一以导航 `entry.key` 为基准

### 2. 扩展纯 CSS 图标系统

在 [static/css/style.css](/Users/lewis/Lewis/code/git/MtEmployeeAttendance-System/static/css/style.css) 中补齐缺失的 `.icon-*` 样式，继续沿用 `::before` / `::after` 组合绘制。

实现原则：

- 优先复用现有 `.app-side-icon` 盒模型与激活态样式
- 已有图标只做必要微调，不整体推翻
- 缺失图标按实际导航 key 补齐，避免菜单出现无图标项
- 如果个别 key 命名与图标语义不完全匹配，只做最小兼容处理，不改导航行为

### 3. 图标语义规则

图标统一采用企业后台线性描边风格，尽量使用抽象但稳定的业务隐喻：

- 查询类：表格、异常标记、打卡记录、部门工时、下载
- 主数据类：账套、人员、部门、班次、账号
- 修正类：修正单、年休、加班、历史维护

风格规则：

- 默认态低对比，不压过菜单文字
- 激活态只跟随当前菜单高亮，不增加额外装饰
- 图形重心保持居中，避免某些图标明显上飘或下坠
- 线宽和留白统一，避免不同菜单项“像来自两套系统”

## 覆盖范围

本次覆盖“侧边菜单所有页面项”，以实际导航数据输出的 `entry.key` 为准。

预期覆盖大类包括：

- 查询中心相关页面
- 账套中心与主数据页面
- 修正中心与管理维护页面

如果扫描后发现还有额外页面 key，也一并补齐，确保没有遗漏项。

## 验收标准

- 侧边菜单每个页面项都有图标。
- 图标与页面语义基本匹配，不出现明显误导。
- 整体风格统一，没有某几个图标尺寸、粗细或位置明显异常。
- 现有模板测试与导航相关页面渲染不受影响。
- 至少在 `查询中心`、`账套中心`、`修正中心` 三类模块下做一次本地视觉检查。

## 风险与控制

风险：

- 导航 key 命名可能不完全规则，导致图标类名无法直接命中
- 纯 CSS 图标表达力有限，个别业务图标可能需要做取舍

控制方式：

- 先扫描实际 `entry.key`，再补样式，避免盲改
- 图标设计优先统一性，其次才是个别图形细节
- 如遇难以表达的页面，优先选择更通用的业务隐喻，而不是硬凑复杂造型
